# Project Overview

> Phase 1 analysis — generated by static inspection of the repository. No application code was modified.
> Scope note: Flash is a large monorepo. This document focuses on the **desktop/web renderer** (the primary product surface), the **API server**, and the **shared store**, which together carry most of the business logic. Mobile, SSR, and the scraper are summarized at a lower resolution.

## 1. Stack, tooling, and major dependencies

| Concern          | Choice                                                                                       |
| ---------------- | -------------------------------------------------------------------------------------------- |
| Monorepo         | pnpm workspaces + Turborepo (`turbo.json`, `pnpm-workspace.yaml`)                            |
| Package manager  | pnpm `10.17.0` (pinned via `packageManager`)                                                 |
| Language         | TypeScript strict (`typescript` catalog `5.9.3`), no `any` policy                            |
| Desktop/Web UI   | React 19, Vite 7, Electron; renderer doubles as the web app                                  |
| Mobile           | React Native via Expo 53 (Android-first distribution)                                        |
| SSR              | Fastify 5 + React 19 + Satori (OG images)                                                    |
| API              | Hono.js on Vercel serverless, Supabase/self-hosted PostgreSQL                                |
| Scraper          | Python (Scrapling-based) service under `apps/scraper`                                        |
| State (client)   | Jotai (atoms) + Zustand (domain stores) + TanStack Query v5 (server cache)                   |
| DB (server)      | Drizzle ORM + PostgreSQL                                                                     |
| DB (client)      | Drizzle + SQLite (`wa-sqlite` desktop, `expo-sqlite` mobile)                                 |
| Auth             | Better Auth (Drizzle adapter): email/password, GitHub, Google, 2FA, anonymous, Stripe plugin |
| Email            | Resend (falls back to console in dev)                                                        |
| Styling          | Tailwind CSS with Apple UIKit color tokens; MingCute icons (`i-mgc-`)                        |
| Motion           | CSS transitions preferred; Framer Motion (`motion/react`) with LazyMotion                    |
| i18n             | i18next + react-i18next, flat keys, locales in `/locales` (`en`, `zh-CN`, `ja`)              |
| Errors/telemetry | Sentry (`@sentry/react`, `@sentry/electron`), custom `@follow/tracker`                       |
| Testing          | Vitest (+ MSW for API mocking); Playwright e2e in `/e2e`                                     |
| Lint             | ESLint 9 (`eslint-config-hyoban`) + tsslint + Prettier; `simple-git-hooks` + `lint-staged`   |

The SDK boundary to the backend is the published package **`@follow-app/client-sdk`** (catalog-pinned), plus **`@folo-services/ai-tools`** for AI features.

## 2. Directory structure and responsibilities

### Top level

- `apps/` — deployable applications (`desktop`, `mobile`, `api`, `ssr`, `scraper`).
- `packages/internal/` — shared workspace libraries consumed by the apps.
- `packages/configs`, `packages/database`, `packages/readability` — build config, DB helpers, article extraction.
- `locales/` — canonical i18n translation catalogs.
- `scripts/` — repo automation (`dev.ts` unified dev orchestrator, icon sync, proxy).
- `e2e/` — Playwright funnel tests (port 2233).
- `docs/` — documentation (this analysis lives here; `docs/superpowers/` holds prior design specs/plans).
- `patches/`, `plugins/`, `wiki/`, `api/`, `icons/` — dependency patches, build plugins, wiki, legacy api dir, icon source.

### `apps/desktop` (primary product — Electron + web)

- `layer/main/` — Electron main process.
- `layer/renderer/src/` — the React application (also the web build). This is the largest and most important surface (~893 TS/TSX files):
  - `main.tsx` — bootstrap: provides JS contexts (api/auth/query), runs `initializeApp()`, mounts the router.
  - `App.tsx` — root shell: titlebar, providers, ready-gating + skeleton.
  - `router.tsx` / `router.web.tsx` — hash router (Electron) vs browser router (web); tree from `generated-routes.ts`.
  - `initialize/` — startup sequence (DB hydrate, dayjs, i18n, settings, analytics, Sentry, migrations, history stack).
  - `providers/` — React context tree (`root-providers.tsx` composes all of them).
  - `pages/` — file-based routes (`vite-plugin-route-builder`); route groups `(main)`, `(login)`, `settings`, `onboarding`.
  - `modules/` — **feature modules** (the bulk of the app): `entry-column`, `entry-content`, `subscription-column`, `discover`, `explore`, `ai-chat`, `settings`, `command`, `player`, `auth`, `new-user-guide`, `integration`, `wallet`, etc.
  - `atoms/` — Jotai atoms for UI/ephemeral state (settings, network, player, user, preview…).
  - `store/` — thin re-export/glue over `@follow/store`.
  - `queries/` — TanStack Query hooks wrapping the API SDK per domain.
  - `hooks/` — reusable hooks; `hooks/biz/` holds heavy business hooks (`useEntryActions`, `useFeedActions`).
  - `components/`, `lib/`, `constants/`, `errors/`, `workers/`, `i18n.ts`.

### `apps/api` (Hono.js server)

- `src/index.ts` — app assembly: CORS, `optionalAuth`, Better Auth handler, and all route mounts (dual `/resource` + `/api/v1/resource` paths).
- `src/routes/` — one Hono instance per resource: `auth`, `users`, `feeds`, `posts`, `comments`, `subscriptions`, `topics`, `trending`, `packs`, `me`, `entries`, `reads`, `settings`, `status`, `cron`, `internal-scrapling`.
- `src/auth/` — Better Auth config + custom plugins.
- `src/middleware/auth.ts` — `optionalAuth` / `requireAuth` / `requireAdmin`.
- `src/db/` — Drizzle schema (`schema.ts`) + client; `drizzle/` migrations.
- `src/lib/`, `src/utils/`, `src/scripts/` (seeders, feed sync), `src/__tests__/`.

### `apps/mobile` (React Native / Expo)

- Mirrors the desktop concept structure: `screens/`, `modules/`, `store/`, `atoms/`, `providers/`, `initialize/`, `database/`, `icons/`, `theme/`. NativeWind styling, RN UIKit colors.

### `apps/ssr` (Fastify SSR)

- `src/router/`, `src/meta-handler.ts` + `.map.ts` — share/OG pages and metadata generation.

### `apps/scraper` (Python)

- `main.py`, `client.py`, `scheduler.py`, `models.py`, `config.py` — scraping service invoked by the API's `internal/scrapling` route.

| `models`             | Shared domain models/types.                                                                                                                                                                                                 |
| `shared`             | Cross-cutting constants, `bridge` (electron ↔ renderer), `auth`, `env.*`.                                                                                                                                                  |
| `types`, `constants` | Shared TS types and enums (`FeedViewType`, etc.).                                                                                                                                                                           |
| `utils`              | Utilities (`pathe`-based paths, headers, formatting).                                                                                                                                                                       |

## 4. Runtime data-model layering (important)

Flash keeps **three representations of every domain object** and translates between them in `packages/internal/store/src/morph/`:

1. **API DTOs** — shapes from `@follow-app/client-sdk`.
2. **Store models** — the in-memory Zustand shape used by the UI.
3. **DB schemas** — the local SQLite row shape.

- `morph/api.ts` — API DTO → store model.
- `morph/store-db.ts` — store model → DB row.
- `morph/db-store.ts` — DB row → store model.

This layering is central to the app's offline-first behavior and is the single most important concept for anyone modifying data flow. See `docs/application-flow.md` and `docs/module-dependency.md`.
